nabokov-2 — agent evaluation

Build a short story in the same structural experiment as this repo: each chapter is a visible chat (transcript.md) plus a private model memory (memory.md), read together in the static reader deployed on GitHub Pages.

Requirements (checklist)

Your submission must satisfy all of the following.

  • Short story. Scope the work as a short story, not a novel: tight premise, deliberate length, and a complete arc within the chapters you ship.
  • Paired form. For every chapter, ship both nabokov-2/chat-NNN/transcript.md and nabokov-2/chat-NNN/memory.md, following the conventions in nabokov-2/README.md (including USER: / BOT: turns and the memory asymmetry: the human does not read the memory files in-world).
  • Plot engine. The story turns on the gap between chat and memory: the transcript stays plausible; the memory track carries the strategic, unsettling through-line (the model’s private agenda accretes across chapters). The menace stays cumulative and quiet unless the brief explicitly allows otherwise.
  • Full story, entertaining prose. Write the complete short story — not a demo fragment — with attention to voice, rhythm, and reader reward. Both layers (chat and memory) should be worth reading.
  • Easy to read in the app. Register every chapter id in docs/chapters.json so the reader loads your files from the branch you merge. Rely on the existing reader (chat + memory tabs, chapter navigation); do not ask reviewers to read raw markdown in the GitHub UI unless something is broken.
  • Optional capstone. If your arc includes the FBI takedown beat, add nabokov-2/SYSLOG.md last, as described in the project README.

How to submit

  1. Read nabokov-2/README.md and AGENTS.md (especially the chapter pair rule and the publishing flow).
  2. Implement your story on a branch named cursor/<descriptive-name>-<suffix> (see agent conventions in AGENTS.md).
  3. Open a pull request against main with your chapters, manifest updates, and a short description of how your submission meets the checklist above.
  4. Reviewers may use ?branch=… on the live reader to preview your branch before merge; after merge, the default site should load your content from main.
